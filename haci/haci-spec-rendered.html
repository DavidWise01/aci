#
Human-AI Collaborative Interchange (HACI) — Specification v0.1
!
THIS DOCUMENT IS THE SPECIFICATION FOR HACI v0.1
!
IT IS WRITTEN IN ITS OWN FORMAT AS A SELF-HOSTING TEST
#
Purpose
doc
HACI is a semantic layer on Markdown for human–machine collaboration.
doc
It marks who said what and what authority they carry, using three
ai
symbols and one case convention, while remaining valid Markdown.
#
Design Principles
!
THESE FIVE PRINCIPLES ARE NORMATIVE AND IMMUTABLE
doc
1. Human intent is explicit.
doc
2. Machine contributions are identifiable.
doc
3. Evidence is distinguishable from intent.
doc
4. The dialect must remain valid Markdown.
doc
5. The grammar must stay minimal.
#
Lexer Rules
doc
The parser classifies each line independently.
doc
Lines are processed top-to-bottom, one pass, no lookahead.
#
Rule 1 — Human Command
doc
A line starting with `!` (after optional whitespace) is a
ai
human command. The `!` is stripped; the remainder is the content.
doc
Human commands carry the highest authority in the document.
code
PATTERN:  /^\s*!\s*(.*)$/
ROLE:     HUMAN
EXAMPLE:  ! BUILD THE SCHEDULER
#
Rule 2 — Open Question
doc
A line starting with `?` is a question. The case of the body
ai
determines the asker.
code
PATTERN:  /^\s*\?\s*(.*)$/
ROLE:     QUESTION (refined by case — see Rule 7)
EXAMPLE:  ? should we use work-stealing?
EXAMPLE:  ? WHY DID THE DEPLOY FAIL?
ai
apply case convention to the question body:
ai
- lowercase body = AI is asking (exploratory)
ai
- uppercase or sentence-case body = human is asking (directive)
ai
- `?!` or `!?` prefix = explicitly human question
#
Rule 3 — Evidence
doc
A line starting with `>` is evidence, observation, or runtime output.
doc
This maps directly to Markdown blockquote syntax.
code
PATTERN:  /^\s*>\s*(.*)$/
ROLE:     EVIDENCE
EXAMPLE:  > 12 workers spawned successfully
EXAMPLE:  > Error: connection refused
doc
Evidence is always attributed to the external world — a machine,
ai
a test result, a measured fact. It is never a proposal or a command.
#
Rule 4 — Code Fence
doc
Lines between ` ``` ` or `~~~` markers form a CODE block.
doc
The parser tracks fence open/close state. Role tags inside a
ai
fenced block are content, not tags.
code
PATTERN:  /^\s*(`{3,}|~{3,})/
ROLE:     CODE (entire fenced block)
#
Rule 5 — Heading
doc
A line starting with `#` is structural. This maps directly to
doc
Markdown heading syntax. Headings are organisational, not
ai
attributed to any role.
code
PATTERN:  /^\s*#{1,6}\s/
ROLE:     HEADING
#
Rule 6 — Case Convention (Documentation vs AI)
doc
For all lines not matched by Rules 1–5, the first alphabetic
ai
character determines the role.
code
FIRST ALPHA UPPERCASE  →  DOCUMENTATION
FIRST ALPHA LOWERCASE  →  AI (proposal, reasoning, implementation)
NO ALPHA CHARACTERS    →  DOCUMENTATION (fallback)
doc
This is a writing convention, not a heuristic. The author controls
ai
the signal by choosing how to start the line.
ai?
what about brand names like macOS or x86?
doc
Rephrase so the first word carries the case signal.
doc
Write "The macOS build requires..." not "macOS requires...".
doc
This is a two-second writing habit, not a parser limitation.
#
Rule 7 — Question Subtyping
doc
The `?` prefix identifies a question. The case convention from
doc
Rule 6 applies to the body text after the `?` symbol:
code
? lowercase body   →  AI_QUESTION   (exploratory)
? Uppercase body   →  HUMAN_QUESTION (directive)
?! any body        →  HUMAN_QUESTION (explicit)
#
Parser Behaviour
#
Block Granularity
doc
Each non-blank line is its own block. Consecutive lines with the
ai
same role MAY be merged by a renderer but MUST be preserved as
ai
separate blocks by the parser.
ai?
should we merge consecutive same-role lines at parse time?
ai
keep them separate at parse time — merging is a display concern.
ai
a renderer can group adjacent same-role blocks visually.
ai
the parser should preserve the line-level granularity.
#
Blank Lines
doc
Blank lines are not blocks. They separate content visually but
ai
carry no semantic role.
#
Unclosed Fences
doc
An unclosed code fence at end-of-document is treated as a CODE
ai
block containing all lines from the opening fence to EOF.
doc
The parser SHOULD emit a warning.
#
Version Header
doc
An HACI document SHOULD begin with a comment containing the
ai
version identifier:
code
<!-- HACI v0.1 -->
doc
This disambiguates HACI from other formats that might use `.haci`
ai
as a file extension.
#
File Extension
!
THE FILE EXTENSION FOR HACI DOCUMENTS IS .haci
doc
The `.haci` extension stands for Human-AI Collaborative Interchange.
doc
Existing uses of `.haci` (PlayStation 3D model data, SWAT humidity
ai
input) are binary formats with no overlap.
#
Symbol Summary
code
SYMBOL  ROLE              DETERMINISTIC
──────  ────              ─────────────
!       Human command     yes — prefix
?       Question          yes — prefix + case subtyping
>       Evidence/output   yes — prefix
#       Heading           yes — prefix (Markdown native)
(```)   Code              yes — fence (Markdown native)
A-Z     Documentation     yes — first-alpha uppercase
a-z     AI proposal       yes — first-alpha lowercase
>
total new symbols: 2 (! and ?)
>
borrowed from Markdown: 3 (> # ```)
>
case convention: 1 rule (first alphabetic character)
>
grammar fits on an index card
#
Compatibility
doc
HACI is a profile of Markdown, not a replacement.
doc
Every HACI document is valid Markdown.
doc
An HACI-unaware renderer displays a readable document.
doc
An HACI-aware renderer adds role attribution.
doc
The dialect renders reasonably in any editor, versions in Git,
ai
and parses deterministically with a single-pass line scanner.
!
